## 2. v2 五大假设 + Falsifiable Threshold

> 每个假设必须有"用什么数据 / 多少样本 / 什么阈值"才能裁决。
> 不写阈值的不是假设，是愿望。

### H1（命门）：付费意愿 ¥1000/月

- **假设**：3 场补充访谈中，**≥ 2 场**接受 ¥1000/月（或经过价值锚定后落到 ≥ ¥500/月）
- **验证方法**：访谈 Q15-v2 阶梯定价问法
  - "如果一个工具帮你**每月省 30 小时**，从选题到发布全自动，**¥500/月**用不用？"
  - "**¥1000/月**呢？"
  - "**¥2000/月**呢？为什么？"
- **裁决阈值**（D22 已签字 · 不存在 "¥500 继续 B" 的中间路径）：
  - 🟢 ≥ 2/3 接受 ¥1000 → H1 PASS，B 方向立项
  - 🟡 ≥ 2/3 接受 ¥500 但 < 2/3 接受 ¥1000 → **H1 SOFT-FAIL · 直接切 C 选题工具方向**（理由：¥500 ARPU − ¥300-900 视频生成成本 = 负毛利）
  - 🔴 ≤ 1/3 接受 ¥500 → H1 HARD-FAIL，**v3 项目暂停**，回退到原始 Discovery 重审赛道

### H2：工作流 UI 形态正确

- **假设**：用户看到 liblibtv 风格节点编排 demo（用 Figma / 截图代替）后，**≥ 2/3** 表达"对，就是这种"
- **验证方法**：访谈中展示 liblibtv 截图 + 自画 mockup（选题 → 脚本 → 分镜 → 视频 → 发布 5 节点串联）
- **追问**：你希望节点之间能"插手编辑"还是"全自动跑完"？（决定 UI 复杂度）
- **裁决阈值**：≥ 2/3 表达"对" + 至少 1 人能说出哪些节点想插手 → PASS

### H3：自动发布需求强度（决定 MVP-1 是否含发布）

- **假设（修订 · 与 D25 默认对齐）**：用户**接受** MVP-1 不含发布，自己手动发或用现有脚本
- **验证方法**（题序很重要，不要先卖"全自动"）：
  - "你现在发布是手动还是脚本？具体怎么操作？"
  - **"如果工具只导出剪映工程文件 / 脚本文本，你自己手动发或用现有方法发布，¥1000 还付吗？"**（关键问题，验证 D25 (c) 默认）
  - "如果工具能合规走抖音 OAuth（有平台审核 + 每日上限），你愿意每月多付 ¥200 吗？"
  - "如果走全自动模拟点击但账号可能被抖音封号，你接受吗？"
- **裁决阈值**：
  - 🟢 ≥ 2/3 接受"导出 + 手动发"+ ¥1000 仍付 → D25 (c) 锁定，MVP-1 不做发布
  - 🟡 ≥ 2/3 坚持要 OAuth 合规接管 + 愿意加价 → MVP-2 评估 (a)
  - 🔴 ≥ 2/3 说"必须全自动模拟点击" → 警示！项目要么接受封号风险（不推荐），要么砍掉这部分用户

### H4：批量生产是真需求

- **假设**：用户实际会一次性生成 5+ 条视频变体，**而不是**一次只做 1 条
- **验证方法**：
  - "你现在一天 / 一周大概想发几条？"
  - "如果工具一次生成 5 个不同钩子的版本让你挑 1 个，有用吗？"
- **裁决阈值**：
  - 🟢 ≥ 2/3 表达"会跑批量" → 工作流引擎设计为多任务并行
  - 🔴 < 2/3 → MVP 砍批量，单任务串行

### H5：选题模块价值

- **假设**：每天给用户 5 条"今日抖音/小红书 trending + 为什么火 + 你能怎么改造"，用户**愿意优先用这个选题**而不是自己刷
- **验证方法**：
  - "你现在选题大概花多久？" → 量化每周时长
  - "如果工具每天早上推 5 个选题，你会用还是自己刷？"
- **裁决阈值**：
  - 🟢 ≥ 2/3 说"会用" → 选题模块进 MVP
  - 🟡 ≥ 2/3 说"会参考但还是自己刷" → 选题降级为 v3.5 加 module
  - 🔴 ≥ 2/3 说"我自己选题更准" → 选题不做，从工作流第 1 步直接接"用户输入选题"

4. **¥1000/月**呢？

### H2 验证：工作流形态（5 min）
