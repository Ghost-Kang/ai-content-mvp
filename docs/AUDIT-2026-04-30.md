## 🔴 严重（CRITICAL — 安全 / 合规 / 财务）

### #1 多租户 RLS 实际未启用（纵深防御缺失）

**现状**
- `src/server/context.ts:7-13` 注释自陈：Drizzle 用 `postgres` 超级用户连库，**所有 RLS 策略被 bypass**。
- 任何 `SELECT/UPDATE` 漏 `eq(table.tenantId, ctx.tenantId)` 即跨租户数据泄露。
- 没有任何代码调用 `set_config('app.tenant_id', ...)`，session var 永远为空 —— 即使迁到非超级用户角色也立刻全表 0 行。

**修复路径**
1. 在 Supabase 建独立 `app_user` 角色（GRANT 表级 SELECT/INSERT/UPDATE/DELETE，不给 BYPASSRLS）
2. 准备第二条连接串 `DATABASE_URL_APP`，端口走 pgbouncer
3. `createContext` 里给每个事务 `SET LOCAL app.tenant_id = '<uuid>'`
4. 切换前用 `scripts/probe-cross-tenant.ts` 跑一遍验证

**本次状态**：⏳ 准备中 —— 见 `docs/RLS-CUTOVER.md`，迁移 SQL + 代码 helper 已就位，需运维在 Supabase 执行 GRANT/REVOKE 后切环境变量。

## 🟠 高（HIGH — 真 bug，影响用户/数据）

### #4 月度用量重复累加（resume 路径） ✅ 已修复

**现状**
- `src/lib/workflow/orchestrator.ts:178 + 254 + 313`
- hydrate-and-skip 把已 done 的 step `costFen` 计入 `totalCostFen`
- 节点失败时 `bumpMonthlyUsage(totalCostFen)` 把 hydrated 部分**再算一次**
- 一次成功跑完后 line 313 再 bump 一次

跑 N 次 retry 后，`monthly_usage.total_cost_fen` 可能是真实消耗的 1.5×–2×。方向是 over-count（用户更早触顶），不是财务漏洞，但 admin dashboard 数字失真。

**修复**：bump 改为只对 "本次 dispatch 内新执行" 的节点累加；hydrated 节点不再 bump。
**本次状态**：✅ 修复 in commit。
